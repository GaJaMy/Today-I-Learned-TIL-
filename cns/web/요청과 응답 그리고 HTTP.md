# 요청과 응답, HTTP
## HTTP와 HTTPS
- HTTP는 평문 형태의 데이터
- HTTPS는 TLS를 통해 데이터를 암호화 하여 통신
- 웹에서 클라이언트와 서버가 요청과 응답을 통해 데이터를 주고받기 위한 애플리케이션 계층 프로토콜
- 일반적으로 TCP/IP 위에서 동작하며, HTTP/3는 UDP 기반의 QUIC을 사용
- 무상태(Stateless) 프로토콜
    - 서버는 기본적으로 이전 요청의 상태를 기억하지 않음
    - 쿠키, 세션, 토큰 등을 사용해 로그인 상태 등을 유지할 수 있음
- 요청 메시지는 주로 메서드, URL, 헤더, 본문으로 구성
- CRUD 
    - Create / Read / Update / Delete
    - 웹 서비스에서 자원을 생성·조회·수정·삭제하는 기본 동작

### 헤더
- 요청 또는 응답에 대한 부가 정보를 전달
- 데이터의 형식, 인증 정보, 캐시 정책, 클라이언트 정보 등을 포함
- 이름: 값 형식으로 작성
```
    Content-Type: application/json 
    Authorization: Bearer access-token 
    Accept: application/json
```
- 대표적인 헤더:
    - Content-Type: 본문 데이터의 형식
    - Authorization: 인증 정보
    - Accept: 클라이언트가 받을 수 있는 응답 형식
    - Cookie: 클라이언트가 서버에 전달하는 쿠키
    - Cache-Control: 캐시 사용 정책

### 본문
- 서버에 전달하거나 서버로부터 전달받는 실제 데이터
- 요청과 응답 모두 본문을 가질 수 있지만, 항상 존재하는 것은 아님
- JSON, HTML, 파일, 이미지 등의 데이터를 포함할 수 있음
```
    { 
        "name": "서형동", 
        "email": "example@email.com" 
    }
```

### 메서드
- 멱등성 
    - 같은 요청을 여러 번 실행해도 서버 리소스의 최종 상태가 한 번 실행했을 때와 같은 성질
    - 응답 내용이나 상태 코드는 달라질 수 있음
- GET - 자원의 조회, 멱등성 O
- POST - 자원의 생성, 멱등성 X
- PUT - 자원의 전체를 업데이트, 완전 대체, 멱등성 O
- PATCH - 자원의 일부를 업데이트, 일부 수정, 멱등성 X
- DELETE - 자원을 삭제, 멱등성 O

### URL
- 요청할 자원의 위치를 나타내는 주소
- REST API에서는 URL에는 자원을 명사로 표현하고, 수행할 동작은 HTTP 메서드로 구분하는 것이 일반적
- 프로토콜, 도메인, 포트, 경로, 쿼리 파라미터 등으로 구성
    ```
    https://api.example.com:443/users/1?detail=true
    ```

    - https: 프로토콜
    - api.example.com: 도메인
    - 443: 포트 번호
    - /users/1: 자원의 경로
    - detail=true: 쿼리 파라미터

### 응답 상태 코드
- 1XX (정보): 요청을 받았으며 처리가 계속 진행 중임을 나타냄.
- 2XX (성공): 클라이언트의 요청이 정상적으로 처리되었음을 나타냄.
- 3XX (리다이렉션): 요청을 완료하려면 다른 위치로 이동하거나 추가 작업이 필요함을 나타냄.
- 4XX (클라이언트 오류): 잘못된 요청, 인증 실패 등 클라이언트 측 문제로 요청을 처리할 수 - 없음을 나타냄.
- 5XX (서버 오류): 서버 내부 문제로 정상적인 요청을 처리하지 못했음을 나타냄.
