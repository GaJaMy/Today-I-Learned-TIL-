# AWS
```
AWS는 아마존닷컴의 클라우드 컴퓨팅 사업부(Amazon Web Services)를 뜻함
물리적인 서버를 직접 구축할 필요 없이, 인터넷을 통해 필요한 만큼의 IT 인프라(서버, 데이터베이스, 스토리지 등)를 빌려 쓰고 사용한 만큼만 요금을 지불하는 대표적인 클라우드 서비스
```

## Region과 Availability Zone

### Region
```
AWS 데이터 센터가 존재하는 하나의 지리적 지역
```
- 리전 선택 요소
    - 사용자와의 거리 및 지연 시간
    - 서비스 제공 여부
    - 데이터 보관 관련 법률
    - 비용
    - 재해 복구 전략

### Availability Zone
```
AZ는 하나의 Region 안에 존재하는 서로 독립된 물리적 위치
```
- 각 AZ는 하나 이상의 데이터센터로 구성 
- 전력, 네트워크, 연결 설비가 이중화되어 있다. 
- 같은 Region의 AZ들은 빠른 전용 네트워크로 연결되지만, 하나의 장애가 다른 AZ로 번지는 것을 줄이도록 물리적으로 분리
- 서버를 서로 다른 AZ에 배치해야 장애 범위를 분리할 수 있다.
- Multi-AZ
    - 여러 AZ에 자원을 배치하여 하나의 AZ 장애에 대비하는 구성
    - 하나의 AZ의 장애가 생기더라도 다른 AZ의 서버가 동작
    - 이것을 고가용성, High Availability이라고 한다.

## IAM
```
AWS 리소스에 대해 다음 두 가지를 관리하는 서비스

인증(Authentication): 당신이 누구인가?
인가(Authorization): 당신이 무엇을 할 수 있는가?
```

### IAM 구성 요소
- Root User
    - AWS 계정을 처음 만들 때 사용하는 이메일 주소의 계정
    - 해당 AWS 계정의 모든 서비스와 리소스에 접근 할 수 있는 계정
    - 평상시에는 사용하지 않고, 루트 사용자만 수행할 수 있는 일부 계정 관리 작업에만 사용 권장
    - MFA를 설정하고 루트 사용자의 엑세스 키는 만들지 않는 것을 기본 보안 원칙으로 한다.
- IAM User
    - AWS 계정 안에서 생성하는 개별 사용자
    - 사람 또는 애플리 케이션을 나타내는 영구적인 자격 증명
    - 현재 AWS의 권장 방식은 장기 IAM User 자격 증명을 사용하는 것보다 IAM Identiy Center나 외부 인증을 통해 임시 자격 증명을 사용하는 것
    - 회사 구성원 별로 만들수 있음
    - IAM User 마다 서로 다른 권한을 줄 수 있음
- IAM Group
    - IAM User를 묶어 공통 권한을 관리하기 위한 용도
    - 그룹 자체가 AWS에 로그인하거나 역할처럼 임시 권한을 얻는 것은 아님
- IAM Role
    - 특정 사용자에게 영구적으로 소속되는 계정이 아니라, 필요한 주체가 잠시 맡아서 사용하는 권한 묶음
    - 일반적으로 고정된 비밀번회나 장기 Access Key를 부여하지 않음
    - 일정 시간 동안 사용할 수 있는 임시 자격증명이 발급
    - EC2가 S3에 접근하는 예시
        - EC2가 S3에 접근하려면 AWS API를 호출할 권한이 필요
        - Access Key를 코드에 직접 저장하게 되면? ==>> github이나 로그를 통해 키가 유출될 위험이 크다.
        - EC2에 IAM Role을 연결 시킴으로서 Spring Boot Application이 S3에 접근 가능!
 
    | 구분    | IAM User         | IAM Role            |
    | ----- | ---------------- | ------------------- |
    | 대상    | 특정 사용자 또는 애플리케이션 | 사용자, AWS 서비스, 다른 계정 |
    | 소유 방식 | 특정 대상에게 고정       | 필요한 주체가 맡아서 사용      |
    | 자격 증명 | 장기 자격 증명 사용 가능   | 임시 자격 증명 사용         |
    | 비밀번호  | 설정 가능            | 자체 로그인 비밀번호 없음      |
    | 대표 사례 | 개발자 계정           | EC2가 S3에 접근         |
    | 권장 용도 | 제한적인 개별 사용자 관리   | AWS 서비스와 워크로드 권한    |

- IAM Policy
    - 어떤 작업을 허용하거나 거부할지를 작성한 JSON 문서
    - IAM User, Group, Role 또는 일부 AWS 리소스에 연결해 권한을 정의
    - AWS는 요청이 들어올 때 적용 가능한 정책을 평가하여 허용 여부를 결정
    - 정책 구성 요소
        - Effect : 허용할 지 거부할지를 설정 (Allow 또는 Deny)
        - Action : 어떤 작업을 허용할지 결정 ("s3:GetObject" 등 리소스마다 다름)
        - Resource : 어떤 AWS 리소스에 대해 작업할 수 있는지를 지정
        - ARN : AWS 리소스를 식별하는 고유한 이름

### IAM 권한 판단 규칙
1. 기본적으로 모든 요청은 거부된다.
2. 정책에서 명시적으로 Allow해야 허용된다.
3. 하나라도 명시적인 Deny가 있으면 최종적으로 거부된다.

- 다음 두 정책이 동시에 적용되었다고 가증.
    ```
    정책 A
    → S3 전체 접근 Allow

    정책 B
    → 중요 버킷 삭제 Deny
    ```
- 사용자가 중요 버킷을 삭제하려 하면 Allow + Deny → 최종 결과 Deny

### 최소 권한 원칙
```
IAM에서 가장 중요한 보안 원칙은 최소 권한 원칙(Principle of Least Privilege)

사용자가 업무에 필요한 권한만 가져야 한다는 의미
```

- 처음부터 넓은 권한을 주는 것보다 필요한 최소 권한부터 부여한 뒤 필요한 권한을 추가하는 방식을 권장
